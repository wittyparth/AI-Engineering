# Cornerstone Project: Multi-Provider LLM Gateway

**Time**: 2-3 days  **Difficulty**: Medium  **Portfolio**: Yes

## The Problem

You're building AI features for a product. You don't want to hardcode a single provider. You want a unified API that can route to any LLM provider, handle failures gracefully, and track costs.

## Requirements

### Must Have
- [ ] FastAPI service with `/chat` and `/chat/stream` endpoints
- [ ] Support OpenAI, Anthropic, and Ollama (local)
- [ ] Automatic provider fallback on failure
- [ ] Streaming via SSE
- [ ] Token counting and cost tracking per request
- [ ] Configurable via environment variables
- [ ] Rate limiting (configurable RPM)
- [ ] Request/response logging

### Should Have
- [ ] Request caching (exact same prompt → cached response)
- [ ] Health check endpoint returning provider status
- [ ] Structured output support (JSON mode)
- [ ] OpenAPI docs (auto-generated by FastAPI)
- [ ] Graceful degradation (return partial results?)

### Nice to Have
- [ ] Admin UI showing provider health + cost dashboard
- [ ] A/B routing (route X% of traffic to different models)
- [ ] Request queue with prioritization
- [ ] Prompt template management

## Architecture

```
Client → FastAPI Gateway → Rate Limiter → Router → Provider 1
                                           ├── Provider 2 (fallback)
                                           └── Provider 3 (local)
                 ↓
            Telemetry → Cost Logger → Prometheus
```

## Design Decisions

You must decide (and justify in your design doc):

1. **Sync vs async endpoints** for non-streaming? (Consider: FastAPI async for I/O, sync for CPU)
2. **How to detect provider failure**? Timeout? HTTP status? All 3?
3. **Fallback strategy**: failover to next provider or retry same provider first?
4. **Cache invalidation**: how long does cache live? Per-user or global?
5. **Rate limit scope**: per-user or global? What happens when limit is hit?
6. **Cost tracking granularity**: per-request, per-user, per-provider, per-model?

## 🔴 Senior Design Notes

- **Don't cache streaming responses** — you lose the token-by-token experience
- **Rate limit at the gateway, not just the client** — protects backend providers
- **Fallback ≠ failover** — fallback tries same request on another provider, failover is for complete provider outage
- **Cost tracking must be async** — don't block the response path for telemetry
- **Provider credentials in secrets manager** — never in code or env vars in production

## Code Review Rubric

| Criteria | Pass | Fail |
|----------|------|------|
| Architecture | Clean separation router/service/client | All logic in route handlers |
| Error handling | All error types handled, graceful degradation | Uncaught exceptions, silent failures |
| Streaming | Works end-to-end, handles disconnects | Streams but errors on disconnect |
| Testing | Unit + integration tests for fallback | No tests |
| Config | Environment-based, pydantic-settings | Hardcoded values |
| Observability | Logging + cost tracking + health | No monitoring hooks |
| Performance | Handles concurrent requests | Sequential only |
| Security | Input validation, rate limiting | No validation |
